# 结构化文档模型 V1 方案

## 1. 方案目的

本方案用于指导 LAYOUT3.0 第二阶段的主链路纠偏与持续开发。

当前 M1 已经打通 Markdown 输入、解析、分页预览、文件管理和 PDF 导出最小链路，但当前实现仍偏向“纯 Markdown 解析结果驱动”。如果继续在这个基础上直接增加字体、段落、高亮、答案挖空等能力，会出现样式无处保存、重新解析后丢失、画布编辑和源码编辑互相覆盖的问题。

因此，后续主链路必须调整为：

```text
Markdown 输入 / 导入
→ 生成结构化文档模型 LayoutDocument
→ 画布、右侧属性面板、分页、导出都消费 LayoutDocument
→ .layout 保存完整模型
→ Markdown 只作为输入源和可选导出格式，不再作为唯一真相源
```

## 2. 核心结论

1. Markdown 是输入层，不是后续所有编辑结果的唯一保存层。
2. `.layout` 是主工程文件，负责保存结构、样式、资源、模板、答案标记和视图状态。
3. 文档模型要借鉴 Word 的“段落 paragraph + 行内片段 run”思想。
4. 一个块内某几个字的字体、颜色、高亮、下划线等格式，必须落到 `TextRun` 级别。
5. 答案挖空不能只依赖红色字体，必须保存为语义标记 `answer`。
6. 红色只是答案显示状态下的样式，隐藏答案和下划线挖空只是渲染模式，不能删除原文。

## 3. 总体模型

```ts
interface LayoutDocument {
  version: string;
  id: string;
  title: string;
  source: string;
  blocks: LayoutBlock[];
  resources: LayoutResource[];
  styles: LayoutStyleSheet;
  template: LayoutTemplateState;
  viewState: LayoutViewState;
  meta: LayoutDocumentMeta;
}
```

字段说明：

| 字段 | 用途 |
|---|---|
| `version` | 模型版本，用于后续迁移 |
| `id` | 文档唯一 ID |
| `title` | 文档标题 |
| `source` | 原始 Markdown 源码，保留输入来源 |
| `blocks` | 真正用于编辑、分页和导出的结构块 |
| `resources` | 图片、附件等资源引用 |
| `styles` | 默认样式、模板样式、字符样式和块样式 |
| `template` | 当前模板和模板覆盖信息 |
| `viewState` | 答案显示模式、缩放、选中状态等 |
| `meta` | 字数、页数估算、更新时间等元信息 |

## 4. 块模型

```ts
type LayoutBlockType =
  | 'paragraph'
  | 'heading'
  | 'list'
  | 'table'
  | 'image'
  | 'equation'
  | 'blockquote'
  | 'code'
  | 'horizontalRule'
  | 'pageBreak';

interface LayoutBlock {
  id: string;
  type: LayoutBlockType;
  sourceRange: SourceRange | null;
  blockStyleRef: string | null;
  blockStyleOverrides: BlockStyleOverrides;
  textRuns: TextRun[];
  pagination: BlockPagination;
  metadata: Record<string, unknown>;
}
```

字段说明：

| 字段 | 用途 |
|---|---|
| `id` | 块稳定 ID，用于画布选中、属性面板、分页定位 |
| `type` | 块类型 |
| `sourceRange` | 对应 Markdown 源码范围，便于后续同步 |
| `blockStyleRef` | 段落、标题、题干、答案解析等块样式引用 |
| `blockStyleOverrides` | 对齐、缩进、行距、段前后距、背景色等块级局部覆盖 |
| `textRuns` | 行内文字片段，支持局部字体、高亮、答案标记 |
| `pagination` | 分页控制，如强制分页、标题跟随下一段 |
| `metadata` | 不同块类型的扩展信息 |

## 5. 行内片段模型

`TextRun` 是本方案最关键的结构。它负责表达“同一个段落里，只有某几个字发生格式变化”。

```ts
interface TextRun {
  id: string;
  text: string;
  sourceRange: SourceRange | null;
  marks: TextMark[];
  charStyleRef: string | null;
  styleOverrides: TextStyleOverrides;
  annotations: TextAnnotation[];
}
```

字段说明：

| 字段 | 用途 |
|---|---|
| `id` | 行内片段 ID |
| `text` | 片段文字 |
| `sourceRange` | 片段对应 Markdown 源码范围 |
| `marks` | 加粗、斜体、下划线、删除线、链接等语义格式 |
| `charStyleRef` | 字符样式引用 |
| `styleOverrides` | 字体、字号、颜色、高亮、字间距等局部视觉格式 |
| `annotations` | 答案、重点、批注、挖空等语义标记 |

示例：只给“重要”两个字加高亮。

```ts
textRuns: [
  { text: '这是一个', styleOverrides: {}, annotations: [] },
  { text: '重要', styleOverrides: { highlightColor: '#fff3a3' }, annotations: [] },
  { text: '知识点', styleOverrides: {}, annotations: [] },
]
```

## 6. TextRun 切分与合并规则

当用户选中一个块里的某几个字并修改格式时，模型层按以下规则处理：

```text
选中几个字
→ 找到所在 TextRun
→ 按选区边界切成前段、选中段、后段
→ 给选中段写入 marks / styleOverrides / annotations
→ 相邻且格式完全一致的 TextRun 自动合并
```

示例：

原始内容：

```text
这是一个重要知识点
```

用户只选中“重要”并设置红色、高亮、加粗：

```ts
textRuns: [
  { text: '这是一个', marks: [], styleOverrides: {}, annotations: [] },
  {
    text: '重要',
    marks: ['bold'],
    styleOverrides: {
      color: '#d92d20',
      highlightColor: '#fff3a3',
    },
    annotations: [],
  },
  { text: '知识点', marks: [], styleOverrides: {}, annotations: [] },
]
```

## 7. 样式优先级

样式优先级固定为：

```text
默认样式
→ 模板样式
→ 块样式 blockStyleRef
→ 块局部覆盖 blockStyleOverrides
→ 字符样式 charStyleRef
→ 行内局部覆盖 TextRun.styleOverrides
```

这样可以同时支持：

- 全局模板统一换风格。
- 段落级设置对齐、缩进、行距。
- 某几个字单独设置字体、字号、颜色、高亮。
- 清除格式时能回退到上一级样式。

## 8. 答案挖空模型

答案不能只靠红色字体判断。红色可能是重点，也可能是答案，所以答案必须保存为语义标记。

```ts
type AnswerDisplayMode = 'show' | 'hide' | 'underline';

interface AnswerAnnotation {
  type: 'answer';
  id: string;
  blankMode: 'underline' | 'hidden';
}
```

答案片段示例：

```ts
{
  text: '二氧化碳',
  styleOverrides: {
    color: '#d92d20',
  },
  annotations: [
    {
      type: 'answer',
      id: 'answer-001',
      blankMode: 'underline',
    },
  ],
}
```

全局显示模式存放在 `viewState`：

```ts
interface LayoutViewState {
  answerDisplayMode: AnswerDisplayMode;
  zoom: number;
  selectedNodeId: string | null;
}
```

渲染规则：

```text
show:      水的化学式是 H2O
hide:      水的化学式是
underline: 水的化学式是 ______
```

注意：

1. 隐藏答案和下划线挖空只是渲染状态。
2. 原始答案文字必须一直保存在 `TextRun.text` 中。
3. 导出教师版、学生版、挖空版 PDF 时，只切换 `answerDisplayMode`。
4. 可以提供“选中文字设为答案”和“将红色文字识别为答案”两个入口，但最终都必须写入 `annotations`。

## 9. Markdown 与 .layout 的关系

| 格式 | 定位 |
|---|---|
| `.md` | 纯 Markdown 输入文件，可导入、可导出 |
| `.layout` | 主工程文件，保存完整模型 |
| `.pdf` | 交付导出结果 |

V1 阶段规则：

```text
Markdown 输入 / 导入
→ 生成 LayoutDocument
→ 后续可视化编辑写入 LayoutDocument
→ 保存工程时写入 .layout
```

源码编辑同步策略在 V1 先保持简单：

```text
用户修改 Markdown 源码
→ 重新生成 LayoutDocument
→ 如已有可视化微调，提示可能覆盖局部样式
```

后续再考虑更复杂的增量同步。

## 10. V1 实施范围

V1 只做数据底座，不急着做完整交互。

本步要做：

1. 新增 `Document Model` 类型定义。
2. 新增 Markdown 到 `LayoutDocument` 的转换函数。
3. 给段落、标题、列表、图片、表格、分页符生成稳定 `id`。
4. 段落和标题生成基础 `textRuns`。
5. `/pagebreak` 进入模型语义，而不是只作为普通文本处理。
6. 预留 `styleOverrides` 和 `annotations` 字段。
7. 支持模型层表达“某几个字有不同样式”。
8. 支持模型层表达“某几个字是答案”。
9. 保持现有分页预览、导出和 M1 冒烟链路不被破坏。

本步不做：

1. 不做浮动工具条。
2. 不做真实字体按钮。
3. 不做答案隐藏 UI。
4. 不做复杂表格编辑。
5. 不做精细分页测量。
6. 不做完整 Markdown 双向同步。

## 11. 验收标准

完成 V1 后必须满足：

1. Markdown 可以生成 `LayoutDocument`。
2. 每个块都有稳定 `id`。
3. 一个段落内可以拆成多个 `TextRun`。
4. 某几个字可以单独拥有字体、颜色、高亮等样式覆盖。
5. 某几个字可以被标记为 `answer`。
6. `/pagebreak` 进入模型语义。
7. 当前预览、分页、导出链路不被破坏。
8. `npm run typecheck` 通过。
9. `npm run build` 通过。
10. `npm run smoke:m1` 通过。
11. `项目开发进度.md` 已同步登记。

## 12. 后续开发顺序

建议顺序：

```text
结构化文档模型 V1
→ .layout 工程文件读写
→ Markdown 导入到模型
→ 画布元素选择与模型定位
→ 文本与段落局部样式编辑
→ 答案标记与显示模式切换
→ 样式覆盖与模板优先级
→ 精细分页二次推进
→ 表格、图片、公式可视化编辑
→ 目录、大纲与页码回填
```

每一步都必须回写 `项目开发进度.md`。
