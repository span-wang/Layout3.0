## 1. 本轮结论

SQLite 技术路线已经通过阶段 0 验证，可以在后续 `PH3-13B` 使用 `better-sqlite3`。RAGFlow 0.25.0 的精确 `document_ids` 能在排序前过滤，但 `document_ids` 非空时会忽略 `metadata_condition`，因此原方案中的远端 `document_ids + status=active` 联合门禁不能按原描述成立。

用户已确认采用路线 A，阶段 0 正式合同收口为：

- 正式检索由 Main 根据 SQLite 当前有效发布关系生成精确 active `document_ids`，交给 RAGFlow 做排序前过滤。
- 允许集合为空时在本地失败关闭；RAGFlow 返回后再按 SQLite 当前有效发布关系二次校验。
- pending 质量通道由有效 `quality_run_id` 解析本次精确 pending IDs，不能接受 Renderer 自选远端 ID。
- `status` metadata 继续用于审计、漂移、人工诊断和升级复测，但不作为 RAGFlow 0.25.0 的联合安全门。
- `PH3-13A` 已完成；不开始 `PH3-13B`，不修改生产知识库，也不解析现有英语影子库，直到用户再次确认下一步边界。

## 4. 不随路线变化的业务契约

以下契约保持不变：

- `canonical_id`、`publication_branch_key`、`version_id`、`content_hash` 和 RAGFlow document ID 不混用。
- 业务状态继续拆成 `workflow_status`、`processing_health`、`index_publication_status`。
- 远端状态值只使用 `pending / active / superseded / archived`。
- 正式检索的允许 ID 只能由 Main 根据 SQLite 当前发布关系生成，Renderer 不能提交任意远端 ID。
- 允许 ID 集合为空时直接返回无结果，禁止省略过滤后搜索全库。
- pending 质量通道只能通过有效 `quality_run_id` 解析精确候选 ID，不能成为长期旁路。
- RAGFlow 返回后必须再次连接 SQLite 发布关系，排除未知、pending、替代、归档和分支不符结果。
- RAGFlow metadata 仍用于审计、漂移扫描和人工诊断，但在 0.25.0 中不能把它当作与 `document_ids` 同时生效的安全门禁。

## 5. 已确认路线与保留的降级方案

### 路线 A：精确 active ID 作为正式前置门禁（已确认）

- SQLite 只从当前有效 `material_publications` 生成正式 `document_ids`。
- 正式网关只把这组精确 ID 交给 RAGFlow，并在空集合时本地失败关闭；`status=active` 继续写入和回读，但不声明其在 0.25.0 中参与联合过滤。
- pending 质量网关使用由有效 `quality_run_id` 从 SQLite 解析出的精确 pending ID；空集合、过期运行或映射不完整同样失败关闭。
- 返回后继续执行 SQLite 二次校验。
- 通过 SQL 约束、状态机测试、网关集成测试和故障注入防止 pending ID 被错误放入正式集合。
- RAGFlow 升级后重新执行联合合同；只有确认 `document_ids + status` 在排序前求交，才可把 status 恢复为额外防御，且不能移除精确 ID 主门禁。

优点是已经通过排序前过滤实测，能保留 SQLite 事务作为唯一发布切换点，索引和发布成本较低。代价是远端 active metadata 只作为审计信息，不能形成第二个远端过滤条件。

### 路线 B：物理 staging/live 数据集代次隔离（保留为降级方案）

- pending 资料只进入 staging 数据集。
- 正式请求只访问 SQLite 当前选择的 live 数据集代次。
- 发布通过构建和切换 live 代次完成，保留旧代次用于回滚。

优点是远端物理隔离更强。代价是发布需要复制或重建索引，资源、时间、补偿和蓝绿治理复杂度明显更高。
