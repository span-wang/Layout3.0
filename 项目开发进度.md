# 项目开发进度

## 1. 文档目的

本文件用于跟踪 `LAYOUT3.0` 的实际开发推进情况，统一记录阶段目标、当前状态、里程碑达成情况、风险与阻塞、以及每一步完成后的变更日志。

进度跟踪以以下文档为唯一基线：

- `LAYOUT3.0产品开发文档.md`
- `功能设计及需求.md`

若需求、技术方案、阶段规划发生变更，应先更新基线文档，再同步更新本文件。

## 2. 跟踪原则

1. 以“可验证的完成结果”作为进度单位，而不是以“正在做”作为完成标准。
2. 每次只推进一个清晰的小步，并在完成后立即登记到本文件。
3. 每个进度项都要能对应到需求、模块、阶段或里程碑，避免出现无法验收的模糊描述。
4. 若出现阻塞、返工、范围变化，必须在进度日志和风险区同步说明。
5. 进度状态优先反映真实情况，禁止为了好看提前标记完成。

## 3. 进度单位定义

“推进一步进度”定义为一次可以独立验收的最小开发推进，至少满足以下条件：

- 有明确对象：功能、模块、文档、测试、脚手架、接口、流程之一。
- 有明确动作：新增、集成、修复、重构、验证、补充文档之一。
- 有明确结果：可以通过代码、文档、测试、命令输出或人工检查确认完成。

符合以下情况时，才可以写入“已完成”：

- 一个模块骨架已创建并可运行。
- 一个功能链路已打通并经过最基本验证。
- 一条接口契约已落地并与调用方对齐。
- 一项文档规范已建立并可供后续执行。
- 一组测试已补齐并验证通过。

以下情况不计为完成一步：

- 只有想法，没有产出物。
- 只做了部分修改，尚不能验证。
- 已开始但仍存在关键缺口。

## 4. 状态定义

| 状态 | 含义 | 使用规则 |
|---|---|---|
| `未开始` | 尚未投入实施 | 仅已识别范围，未产生产出 |
| `进行中` | 已开始推进 | 已有动作，但未达到完成标准 |
| `已完成` | 已达到验收标准 | 必须能说明完成结果与验证方式 |
| `阻塞` | 无法继续推进 | 必须写明阻塞原因、影响范围、下一步 |
| `暂缓` | 当前阶段不做 | 必须说明暂缓依据，如优先级或范围调整 |

## 7. 进度更新规则

每次推进一步后，必须同步更新以下四处内容：

1. 对应阶段拆解表中的 `当前状态`。
2. 对应模块表中的 `当前状态` 或 `说明`。
3. 相关里程碑的 `当前状态`，如果该步影响里程碑判断。
4. 本文件末尾的“进度日志”。

更新时建议使用以下顺序：

1. 先确认该步是否达到完成标准。
2. 再更新状态表。
3. 最后追加一条日志，记录时间、动作、结果、验证方式、下一步。
