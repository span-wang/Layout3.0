# 文件管理功能代码审查报告

## 审查范围

覆盖了文件管理完整链路：`electron/main/file-handlers.ts`、`ipc-handlers.ts`、`preload/index.ts`，前端 `services/FileService.ts`、`DraftService.ts`、`RecentFilesService.ts`，`components/layout/{LeftPanel,AppShell,Toolbar}.tsx`、`components/common/ContextMenu.tsx`，状态层 `store/slices/documentSlice.ts` 及相关类型/工具。

## 1. 完整性问题（功能缺失/契约不齐）

### 1.1 `loadDraft()` 与自动草稿写入是两套独立实现（关键）

- `src/services/DraftService.ts` 已封装 `loadDraft / saveDraft / clearDraft`，具备字段校验、`localStorage` 异常吞掉等规范。
- 但 `AppShell.tsx:155-216` 完全没有用 DraftService，而是**手写两段 `localStorage.getItem/setItem`**，并硬编码键名 `'layout3.draft'`。
- 后果：DraftService 成了死代码；同时两处对 draft 结构的校验标准不一致——`loadDraft` 会校验 `title/source/lastModified` 三个字段类型，而 `AppShell` 的 `JSON.parse(raw)` 只校验了 `source` 字段是否为字符串（`AppShell.tsx:191-194`）。
- 此外 `AppShell.tsx:197` 用魔法字符串 `# 未命名文档\n\n` 去比对"是否与空白文档相同"，但 `constants/workspace.ts:10` 已经把 `starterMarkdown` 改成空串 `''`，这行判断**几乎永远会落入恢复分支**，相当于一个静默 bug 累计器。

### 1.2 `starterMarkdown` 与 `starterMarkdownPlaceholder` 不再配套

- `constants/workspace.ts:10` `starterMarkdown = ''`，而 `EditorPane` / `FileService.createBlankDocument()` 都依赖 `starterMarkdown`。
- 进度日志中多次声称"新文档为 `# 未命名文档`"（如 `AppShell.tsx:197`），但实际写入磁盘是空串。**文档契约、UI 文案、磁盘内容、对比逻辑四处不一致**，归档/最近文件标题也会拿到空字符串然后被 `getBaseName` 兜底成"未命名文档"——属于不直观的隐形行为。

### 1.3 `recentlyOpenedFiles` 在 store 里有，但前端组件从来不用

- `documentSlice.ts:13,85-88` 定义了 `recentlyOpenedFiles` 和 `setRecentlyOpenedFiles`，`AppShell.tsx:49` 却自己用 `useState` 维护了一份 `recentFiles`。
- 后果：store 中的字段永远停留在 `[]`，属于"看似接入全局状态、实际仍为本地 state"；一旦后续接入多窗口/同步，这两套来源会冲突。

### 1.4 `restoreDraft` 已定义但无人调用

- `documentSlice.ts:47-58` 提供 `restoreDraft`，但 `AppShell.tsx:201-207` 走的是 `loadDocument({ title, filePath: null, source })` 路径，把 `filePath` 强制设为 `null`，但同时 `loadDocument` 内部把 `isDirty` 置为 `false`。
- 这意味着"恢复草稿"会**把尚未持久化的草稿当成已保存状态**，下次自动草稿定时器不会触发（依赖 `isDirty === true`），如果此时用户未修改就关闭应用，草稿会被悄悄丢弃。应当走 `restoreDraft` 路径。

### 1.5 文件搜索只匹配文件名、不支持路径/内容搜索

- `LeftPanel.tsx:541-547` 的搜索仅按 `file.name.toLowerCase().includes(...)`，且排除了 `kind === 'directory'`。常量 `searchHints` 已提示"后续支持全文搜索、替换与结果定位"——目前只覆盖了"文件名包含"，进度日志里却描述为"搜索面板"（隐含承诺更高能力），存在能力与文案错位。

### 1.6 重命名/移动对"打开中且被重命名/移动的文档"没有正确同步 store

- `AppShell.tsx:418-451`（`handleRenameEntry`）和 `:483-511`（`handleMoveEntry`）的逻辑只更新了 `filePath`/`title`，**没有重置 `parseState / parseResult / pageLayouts`**。
- 由于 `useMarkdownParser` 依赖 `documentEpoch` 重新触发解析，而 `updateDocumentLocation` 不递增 `documentEpoch`（`documentSlice.ts:65-69`），改名/移动后**预览与编辑器可能停留在旧文档的解析结果上**直到用户主动编辑源码。
- 同时 `recentFiles` 在 rename 流程里被**重复添加**：`updateRecentFilePath` 已经把 `lastOpenedAt = Date.now()` 写进去，紧接着 `:443` 又调了 `addRecentFile(nextActiveFilePath, ...)`，会把刚 rename 完的文件以新路径再插一次到队首，依赖 `addRecentFile` 内部的去重逻辑兜底，但调用次序、副作用规模都让行为不可预期。

## 2. 稳定性问题（运行时/异常路径）

### 2.1 主进程文件操作没有任何 `try/catch` 包裹

- `file-handlers.ts` 中 `rename`、`rm`、`readFile`、`writeFile`、`stat` 全部直接 `await`：
  - `openDocumentAtPath:179` 若文件不存在会抛出 `ENOENT` 原文到渲染层。
  - `renameEntry:316` 同名/已存在时会抛 `EEXIST`；`moveEntry:338` 不校验目标目录是否存在，会抛 `ENOENT`。
  - `deleteEntry:354` 的 `rm(targetPath, { recursive: true, force: false })` 失败会冒泡。
- 渲染层（如 `AppShell.handleOpenRecentFile:391-403`）只有一处 `catch {}` 直接当作"文件已不存在"处理——但**这条路径是猜测**，没有根据 `error.code` 区分，实际可能吞掉权限错误、磁盘 IO 错误，导致用户看不到"删除失败"等真实失败信息。

### 2.2 `getUniqueEntryPath` 存在 TOCTOU 竞态

- `file-handlers.ts:89-102`：先 `stat` 判断存在性，再 `writeFile`/`mkdir`。并发点击"新建 Markdown"时，两个请求会拿到相同的候选路径，第二个会抛 `EEXIST`。建议改为 `fs.open(path, 'wx')` 或 `mkdir(..., { recursive: false })` 让内核原子判定。

### 2.3 `listDirectoryEntries` 的递归读盘会因符号链接/权限错误中断

- `file-handlers.ts:108-149`：递归 `stat`/`readdir` 没有单点 `try/catch`，若用户在默认工作区里放入 `node_modules`、受权限保护的目录、或循环符号链接，整个 `getDefaultWorkspace/openFolder/readDirectory` 都会整体失败。
- 进一步：返回结构里**递归把所有子目录全部展开**（`children: await listDirectoryEntries(entryPath)`），意味着工作区有多大，前端就拿多大的树。工作区稍大（> 几千文件）会直接卡 UI；并且 `getDefaultWorkspace` 在 `app.whenReady` 后**同步阻塞**主进程返回。

### 2.4 `syncWorkspaceRoot` 的 `setCurrentDirectory` 可能产生状态回写

- `AppShell.tsx:104-112`：每次文件操作后都 `refreshDirectory(rootPath, ...)`，并且把 `workspaceRootPath` 显式传回去；store 内部（`documentSlice.ts:75-84`）会用传进来的 `workspaceRootPath` 覆盖现有值——这本身没问题，但配合 `:386-389` 的 `handleOpenEntry` 在打开新文件后也会 `syncWorkspaceRoot`，**每打开一个文件都会触发一次全局 tree 刷新**，对大目录来说就是连续的全量 IO 风暴。

### 2.5 草稿恢复的 effect 依赖被显式注释掉，存在死代码

- `AppShell.tsx:215-216`：`// eslint-disable-next-line react-hooks/exhaustive-deps` 把 `[]` 写死。看似只跑一次，但若 `useAppStore` 在 `loadDocument` 中抛出（例如未来加上 schema 校验），`loadDocument` 内部抛错会**没有任何 try/catch**，React 会把整个 effect 标红并可能影响后续渲染。稳定性与可观察性都欠佳。

### 2.6 草稿 `isDirty` 标志与"内容"判定不一致

- 自动保存 effect（`AppShell.tsx:155-183`）依赖 `isDirty` 启动定时器。但 `restoreDraft` 路径（实际上是 `loadDocument`，见 1.4）会**把 `isDirty` 置为 `false`**——所以一旦"恢复草稿"完成，用户没有任何输入就直接关闭应用，**草稿会被遗忘**。进度日志第 217 条说"自动保存草稿（3 秒防抖）与崩溃恢复提示"，但 1.4 + 2.5 共同形成了草稿链路最薄弱的一环。

### 2.7 `updateRecentFilePath` / `updateRecentFilePathPrefix` 的副作用

- `RecentFilesService.ts:88-118`：rename 后会把 `lastOpenedAt` 强制更新为 `Date.now()`，导致刚刚被重命名的文件在历史列表中跳到第一。语义上 rename 不应当"重新打开"。

### 2.8 `moveEntry` 缺少源路径的同名目标检查

- `file-handlers.ts:325-345`：直接 `path.join(destinationDirectoryPath, path.basename(sourcePath))` 再 `rename`。如果目标目录里已有同名文件/文件夹，会抛 `EEXIST`；而 `renameEntry` 路径会走 `getUniqueEntryPath`（注意：`moveEntry` 实际并没走它），两条路径行为不一致。

### 2.9 `deleteEntry` 不阻止删除受保护/系统目录

- `file-handlers.ts:347-363` 对 `rm(targetPath, { recursive: true })` 没有越界保护。若 `LeftPanel` 的 `isPathWithin` 校验漏掉边界条件（如 `entry.path` 是用户工作区外的路径），就可能误删整盘。建议主进程对 `targetPath` 校验"必须在 `workspaceRootPath` 之内"。

### 2.10 拖拽没有"自包含"防护（前端）

- `AppShell.tsx:548-559` `handleDrop` 只校验 `dragSource !== entry.path`，但允许把一个目录拖到它自己或它的子级（主进程 `moveEntry:333` 有保护，但**用户已经把拖拽目标高亮了**），会出现"拖到自己子目录失败"但 UI 没有任何提示的体验断层。

## 3. 解耦问题（耦合/可测试性）

### 3.1 `AppShell` 是一个 600+ 行的"上帝组件"

- `AppShell.tsx` 同时承担：
  - store 选择（28 个 `useAppStore` 调用）；
  - 文件服务编排（创建/打开/另存为/重命名/删除/移动/刷新工作区）；
  - 草稿定时器与恢复对话框；
  - 状态消息提示；
  - 拖拽/上下文菜单回调；
  - PDF 导出入口。
- 这些职责至少应拆为 `useFileManagement` / `useDraftRecovery` / `useWorkspaceRoot` 三个 hook，便于单测与复用。

### 3.2 `LeftPanel` 把"目录树渲染"与"全局状态/上下文菜单触发"耦合

- `LeftPanel.tsx:458-672` 的 `renderPanelContent` 用大对象参数传递 20+ 回调，签名膨胀。建议抽 `DirectoryTree` / `OutlineList` / `SearchPanel` 三个独立组件，通过 props 注入。

### 3.3 `FileService` 重复声明 `LayoutDirectoryEntry`

- `FileService.ts:22-27` 又声明了一份 `LayoutDirectoryEntry`，与 `preload/index.ts:3-8` 的同名结构同形不同源。没有任何类型桥接；一旦后端字段变化，编译期完全不会发现。应该：
  - 在 `src/types/workspace.ts` 暴露统一类型；
  - 通过 `import type { LayoutAPI } from '../../electron/preload/index'` 拿到 `typeof window.layoutAPI`，让 `requireLayoutApiMethod` 的方法名与返回类型自动联动。

### 3.4 `requireLayoutApiMethod` 通过字符串字面量硬编码 IPC 名

- `FileService.ts:29-40` 的 `LayoutApiMethodName` 联合与 `preload` 中的方法名、IPC 频道没有任何交叉引用。一旦 `preload` 中重命名方法，TypeScript 不会报错。建议改成 `keyof LayoutAPI` 抽取。

### 3.5 `FileService` 内部自行实现 `getBaseName / getDirectoryName`

- `FileService.ts:52-62` 与 `utils/filePath.ts:23-33` 是同一函数的两份实现，且 `getDirectoryName` 与 `getDirectoryNameFromPath` 命名不一致。属于"工具函数被复制"的典型代码异味。

### 3.6 `usePagination` 与 `useMarkdownParser` 共享 store 但没有错误重试入口

- `useMarkdownParser.ts:5-39` 在 `parseError` 之后没有重试入口；用户只能通过再次编辑触发。这两块 hook 都直接读 store 副作用，没有 memo/cache，难以测试。

### 3.7 `AppShell` 的 `handleCreateDocument` 不走 store 的 `resetDocument`

- `documentSlice.ts:20-34` 提供了 `resetDocument`，但 `AppShell:218-230` 选择 `createBlankDocument()` + `loadDocument(...)`。这两个动作对 `parseState / pageLayouts` 的赋值结果虽然一致，但跳过 `resetDocument` 意味着未来 `resetDocument` 增加的逻辑（例如清空最近文件、停止草稿定时器）会被遗忘。

### 3.8 preload 不暴露"订阅/事件"通道

- `preload/index.ts` 只暴露 `invoke` 风格 API，渲染层无法感知"主进程主动通知"事件（如 `app.before-quit`、磁盘文件被外部修改）。当前没有抽象，未来要补"外部修改提示"会改动整个 preload 形状。

## 6. 总结

文件管理主链路在"功能可见性"层面已经基本打通，UI 行为也贴近 Obsidian。但**契约一致性、状态分层、类型共享**这三块偏弱：

- 一是 `DraftService` 与 `recentlyOpenedFiles` 形同虚设，前端组件另起炉灶；
- 二是 `FileService` 和 `preload` 各自维护一份 `LayoutDirectoryEntry`，无类型桥接；
- 三是 `AppShell` 单文件 600+ 行，把 store、IPC 编排、UI 状态、副作用定时器全部塞在一起，**这是后续在 M1 验收中最大的可维护性风险**；
- 四是 `starterMarkdown` / `'# 未命名文档\n\n'` / `'未命名文档'` 三处默认值不一致，是最容易"静默丢数据"的隐藏 bug。

建议下一阶段先以"统一类型、收敛 AppShell、补齐草稿契约"为目标重构，再去做 M1 的端到端验收，否则后续接入 AI 面板和模板库时，文件管理侧的状态/类型债会快速放大。
