# PH2-09 分页算法开发线路

## 1. 文档定位

本文件只服务任务 `PH2-09 精细分页算法（二次推进）`，用于指导分页算法主线的后续设计、试验、实现和验收。

本文件不是项目通用业务基线文档，不替代以下项目级基线：

- `LAYOUT3.0产品开发文档.md`
- `功能设计及需求.md`
- `结构化文档模型V1方案.md`

除非后续明确升级为项目级技术基线，否则其他任务默认不需要读取本文件。

## 2. 当前前提

当前项目已经具备以下分页基础：

- 已完成 `LayoutDocument -> PageLayout[]` 的主链路。
- 已具备分页算法插件注册骨架和统一入口。
- 当前内置两套实验算法：
  - `估算紧凑 V1`
  - `估算平衡 V1`
- 当前算法选择已接入右侧“分页策略”面板，并随 `.layout` 与自动草稿持久化。

这意味着后续分页工作应优先沿“多算法插件并存”的方向推进，而不是再回到单一大函数里叠规则。

## 3. 总体策略

本项目分页算法采用三层策略：

1. 编辑实时层
   目标：用户输入时快速反馈，不明显卡顿。
   主算法：估算型算法。

2. 停顿精修层
   目标：用户停顿后进一步提升分页准确率和版面美观度。
   主算法：真实测量型算法。

3. 导出终态层
   目标：导出和最终校验时尽量接近 Chromium / PDF 最终结果。
   主算法：高保真精排算法。

一句话原则：

实时性和最终精度分层处理，不强求一个算法同时解决所有问题。

## 4. 候选算法池

### 4.1 A 类：估算型分页

适用阶段：

- 编辑实时层
- 快速回填页数
- 大文档快速滚动和初始打开

候选插件：

- `estimated-greedy-v1`
  说明：当前默认算法，优先把当前页尽量放满。
- `estimated-greedy-balance-v1`
  说明：当前实验算法，在页尾溢出时尝试回收最后一块到下一页。
- `estimated-cost-v1`
  说明：在估算基础上引入分页代价函数，优先避免明显差的断点。

优点：

- 实现简单
- 性能好
- 稳定性高
- 适合实时输入

缺点：

- 对真实换行、图片说明、表格内容、公式高度不够准
- 对孤寡行、标题挂靠、页尾美观度的上限有限

### 4.2 B 类：隐藏 DOM 真实测量型分页

适用阶段：

- 编辑停顿后精修
- 页面设置变化后的重新收敛
- 导出前预检查

候选插件：

- `measured-block-v1`
  说明：每个块在隐藏测量容器中按真实样式测量高度，再按块分页。
- `measured-block-cache-v1`
  说明：在 `measured-block-v1` 基础上引入块测量缓存和脏区重排。

优点：

- 和 Chromium 实际渲染更接近
- 容易提升图片、公式、复杂段落的准确率
- 适合逐步替换估算高度

缺点：

- 性能明显弱于纯估算
- 如果整篇全量测量，大文档会卡

### 4.3 C 类：行级真实测量型分页

适用阶段：

- 标题和段落精修
- 孤寡行控制
- 标题与下段同页
- 页尾/页首视觉平衡

候选插件：

- `measured-line-v1`
  说明：对标题、段落、列表项做行级测量。
- `measured-line-orphan-v1`
  说明：在行级测量基础上加入孤寡行规则。

优点：

- 版面效果上限高
- 能解决真正难看的文本断点

缺点：

- 实现最复杂
- 不能直接作为所有块的统一方案

### 4.4 D 类：表格专项分页

适用阶段：

- 表格跨页
- 表头重复
- 表格行拆分控制

候选插件：

- `table-keep-whole-v1`
  说明：优先整表保护，放不下再整体进下一页。
- `table-repeat-header-v1`
  说明：支持跨页并重复表头。
- `table-row-split-v1`
  说明：对超高行做更细粒度拆分。

优点：

- 能把当前最难啃的表格分页单独收口

缺点：

- 需要和通用分页算法协作
- 规则复杂，测试量大

### 4.5 E 类：导出参考型分页

适用阶段：

- 导出回归验证
- 高保真分页对照
- 后续可能的闭源排版引擎接入

候选路线：

- Chromium 隐藏页面精排参考
- Paged.js / Vivliostyle 对照验证
- 商业排版引擎结果对照（如后续需要）

本阶段不直接作为编辑态主引擎，只作为：

- 对照结果
- 验收参考
- 回归基准

## 5. 推荐实施顺序

### 5.1 第一批：继续做估算算法的可比较试验

目标：

- 先把现有插件化主线用起来
- 快速筛出更适合实时编辑的默认算法

建议顺序：

1. `estimated-greedy-balance-v2`
   补强页尾平衡规则。
2. `estimated-cost-v1`
   引入分页代价函数。
3. 脏区重排
   只重排受影响区间，而不是整篇刷新。

验收重点：

- 输入时不卡
- 页数变化稳定
- 常见段落和标题比当前更美观

### 5.2 第二批：引入隐藏 DOM 块级测量

目标：

- 让实际高度替代估算高度
- 提升图片、公式、复杂段落、不同字体下的稳定性

建议顺序：

1. 为标题、段落、代码块做隐藏测量
2. 为图片块做真实高度接入
3. 为公式块做真实高度接入
4. 加块级测量缓存
5. 加脏区失效机制

验收重点：

- 页面设置变化后结果更稳定
- 不同字号/行高/缩进下分页更准确
- 性能仍能接受

### 5.3 第三批：文本行级精修

目标：

- 提升孤寡行、标题挂靠、页尾美观度

建议顺序：

1. 标题与下段同页规则升级
2. 孤行控制
3. 寡妇行控制
4. 段落断点代价函数

验收重点：

- 少出现单独悬空标题
- 段落跨页更自然
- 页尾留白更均衡

### 5.4 第四批：表格跨页专项

目标：

- 把表格从“整块保护”升级为“可跨页但不难看”

建议顺序：

1. 表头重复
2. 行级跨页保护
3. 超高单元格的保守拆分

验收重点：

- 表格不再频繁整体被推走
- 跨页后仍可读
- 表头一致性稳定

## 6. 推荐的插件分层

建议后续把分页插件继续拆成四类接口：

### 6.1 通用分页算法插件

负责：

- 决定分页主流程
- 决定如何扫块
- 决定如何选断点

### 6.2 块测量插件

负责：

- `measure(block, context)`
- 输出真实或估算高度
- 提供可缓存键

### 6.3 断点规则插件

负责：

- `keepWithNext`
- `avoidBreakInside`
- `orphans/widows`
- `tableHeaderRepeat`

### 6.4 分页代价插件

负责：

- 计算某个断点“好不好”
- 用于多候选断点比较

这样做的好处是：

- 后续不是不停新增大而全算法
- 而是算法主流程和规则插件可以组合

## 7. 验证指标

后续所有分页算法试验都建议统一记录以下指标：

### 7.1 稳定性

- 同一份文档连续两次分页结果是否一致
- 同一份文档切换选中态、侧栏状态后分页是否乱跳
- 修改局部内容时未受影响页是否保持稳定

### 7.2 准确率

- 预览分页和最终 PDF 分页差异页数
- 预览分页和最终 PDF 断点差异数
- 高风险块类型的误差：
  - 标题
  - 图片
  - 表格
  - 公式

### 7.3 版面效果

- 孤行标题数量
- 段落单行跨页数量
- 页尾留白极端值
- 表格被整块推走次数

### 7.4 性能

- 首次整篇分页耗时
- 单次编辑后重排耗时
- 大文档滚动时的稳定性
- 测量缓存命中率

## 8. 性能控制原则

分页不能只追求准，还必须能实时编辑。

因此后续实现必须遵守：

1. 默认不整篇全量重排
2. 默认不整篇全量真实测量
3. 优先做脏区重排
4. 优先做块测量缓存
5. 行级测量只用于文本块
6. 高保真算法优先在停顿后或导出前运行

结合当前代码状态，我建议接下来按这个顺序推进：

1. `PH2-09A`
   做 `estimated-greedy-balance-v2`
   目标：继续优化页尾平衡。

2. `PH2-09B`
   做 `estimated-cost-v1`
   目标：把“分页断点好坏”从硬规则升级为代价函数。

3. `PH2-09C`
   做脏区重排
   目标：解决后续算法越来越复杂时的性能问题。

4. `PH2-09D`
   做 `measured-block-v1`
   目标：接入隐藏 DOM 块级真实测量。

5. `PH2-09E`

   目标：处理标题孤行与段落孤寡行。

6. `PH2-09F`

   目标：进入表格跨页专项。

## 10. 当前结论

当前最适合本项目的分页发展方向不是单一算法，而是：

- 估算算法负责实时反馈
- 真实测量负责停顿精修
- 文本行级算法负责版面美观
- 表格专项算法负责跨页难点
- 导出参考链路负责最终校验

后续所有分页小步，优先在这个路线内演进，不再回到“一个大分页函数里继续叠条件”的老路。
