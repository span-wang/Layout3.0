# PH2-09 分页点反馈训练算法设计

## 1. 文档定位

本文件服务任务 `PH2-09 精细分页算法（二次推进）` 的设计小步：

- `PH2-09-pagination-feedback-training-design-v1`

本文档只定义“分页点反馈训练算法”的产品逻辑、数据结构、问题字典和自动优化闭环，不直接替代现有 `分页测试算法1 / max-fill-v1`，也不在本步新增代码实现。

一句话目标：

用户判断每一个分页点的分页情况是否正确，并指出问题；系统把这些反馈转换成训练样本，自动优化后续分页评分参数。

## 2. 核心判断

用户不应该手工指定最终分页点。

正确交互是：

1. 系统先完成一次自动分页。
2. 系统把每个分页点整理成可审核对象。
3. 用户判断该分页点是否正确。
4. 如果不正确，用户指出问题类型。
5. 系统根据反馈自动优化分页规则或评分权重。

也就是说，用户提供的是“这个分页结果哪里不舒服”，不是“你必须在这里断页”。

这样做的好处是：

- 用户不需要理解算法内部细节。
- 系统能从多个反馈中学习偏好规律。
- 后续文档也可以复用这些规律。
- 每一次优化都可解释、可回退、可对比。

## 3. 总体流程

分页反馈训练闭环分为 7 步：

1. 自动分页
   由当前 `max-fill-v1` 生成 `PageLayout[]`。

2. 生成分页点
   系统把每一页的结尾位置转换为 `PaginationBreakReview` 审核对象。

3. 自动预诊断
   系统先标出可疑问题，例如页尾留白过大、标题孤立、段落短尾、表格跳页。

4. 用户审核
   用户对每个分页点选择：`正确`、`不正确`、`不确定`。

5. 用户标注问题
   当选择 `不正确` 时，用户选择一个或多个问题类型，也可以补充备注。

6. 转换训练样本
   系统把用户反馈转换为可计算的训练样本，不直接改原文档结构。

7. 自动优化
   系统根据训练样本调整分页评分权重，重新分页，并给出优化前后对比。

## 4. 分页点审核对象

分页点不是一个单独字符，而是一次分页决策的上下文快照。

建议结构：

```ts
interface PaginationBreakReview {
  breakId: string;
  documentId: string;
  documentFingerprint: string;
  pageNumber: number;
  breakIndex: number;
  algorithmId: 'max-fill-v1';
  before: PaginationBreakSide;
  after: PaginationBreakSide;
  pageMetrics: PaginationBreakMetrics;
  systemDiagnosis: PaginationBreakDiagnosis[];
  userFeedback?: PaginationBreakUserFeedback;
  createdAt: string;
}

interface PaginationBreakSide {
  blockId: string;
  blockType: string;
  textPreview: string;
  fragmentKind: 'whole-block' | 'text-fragment' | 'list-fragment' | 'table-fragment' | 'toc-fragment';
  lineCount?: number;
  remainingLineCount?: number;
}

interface PaginationBreakMetrics {
  pageUsedHeightPx: number;
  pageRemainingHeightPx: number;
  pageFillRatio: number;
  nextPageFirstBlockHeightPx?: number;
  breakMovedDistancePx?: number;
}
```

字段解释：

- `breakId` 用于稳定定位一次分页点审核，不等同于块 ID。
- `documentFingerprint` 用于判断反馈是否还能应用到当前文档版本。
- `before` 表示分页点前一个块或片段。
- `after` 表示分页点后一个块或片段。
- `pageMetrics` 保存当时的页尾空间、填充比例和下一页开头情况。
- `systemDiagnosis` 保存系统自动预判的问题。
- `userFeedback` 保存用户最终判断。

## 5. 用户反馈结构

建议结构：

```ts
type PaginationBreakVerdict = 'correct' | 'incorrect' | 'unsure';

type PaginationProblemSeverity = 'low' | 'medium' | 'high';

interface PaginationBreakUserFeedback {
  verdict: PaginationBreakVerdict;
  problemTypes: PaginationProblemType[];
  severity: PaginationProblemSeverity;
  preferredAction?: PaginationPreferredAction;
  scope: 'current-document' | 'global-preference';
  note?: string;
  reviewedAt: string;
}
```

关键规则：

- `correct` 表示用户接受当前分页点。
- `incorrect` 表示当前分页点需要优化。
- `unsure` 表示本次反馈不进入强训练，只作为低权重参考。
- `scope` 用于区分文档特例和长期偏好。

## 6. 问题类型字典

问题类型需要稳定、中文可解释、可映射到算法动作。

```ts
type PaginationProblemType =
  | 'blank-space-too-large'
  | 'heading-orphan'
  | 'paragraph-short-tail'
  | 'next-page-short-head'
  | 'table-jumped-whole'
  | 'table-cross-page-hard-to-read'
  | 'image-position-bad'
  | 'equation-position-bad'
  | 'column-unbalanced'
  | 'page-jump-too-large'
  | 'manual-break-not-respected'
  | 'content-overflow';
```

| 问题类型 | 中文名称 | 自动识别线索 | 优化方向 |
|---|---|---|---|
| `blank-space-too-large` | 页尾留白过大 | 页尾剩余高度明显大于下一块或下一行高度 | 尝试把下一块、下一行或下一列表项提前 |
| `heading-orphan` | 标题孤立 | 页面最后一个可见内容是标题 | 提高标题与后文同页权重 |
| `paragraph-short-tail` | 段落短尾 | 下一页只承接很短文本或少量行 | 回退到更自然断点，避免短尾 |
| `next-page-short-head` | 下一页开头过短 | 新页开头只有一小段内容后马上结束 | 尝试把更多内容留到下一页或前页 |
| `table-jumped-whole` | 表格整块跳页 | 当前页大空白，下一页整表开始 | 提高表格按行拆分优先级 |
| `table-cross-page-hard-to-read` | 表格跨页难读 | 跨页后缺少表头、行被切得太碎 | 提高表头重复、整行保护或合并行保护权重 |
| `image-position-bad` | 图片位置不合理 | 图片压页底、说明断开或与正文关系异常 | 调整图片前后块绑定和图片换页权重 |
| `equation-position-bad` | 公式位置不合理 | 公式孤立页尾或公式与解释断开 | 调整公式与邻近段落绑定 |
| `column-unbalanced` | 多栏不均衡 | 多栏页面某栏明显过空 | 提高栏内均衡权重 |
| `page-jump-too-large` | 页码跳动过大 | 小编辑导致大量后续断点变化 | 提高稳定性权重，减少无关页重排 |
| `manual-break-not-respected` | 手动分页不符合预期 | `/pagebreak` 附近结果异常 | 强化手动分页优先级 |
| `content-overflow` | 内容溢出 | 块超出正文区或被裁切 | 提高溢出惩罚，强制拆分或换页 |

## 7. 偏好动作字典

用户不必须选择偏好动作，但系统可以从问题类型推导。

```ts
type PaginationPreferredAction =
  | 'move-next-content-back'
  | 'move-current-tail-forward'
  | 'keep-heading-with-next'
  | 'split-table-by-row'
  | 'repeat-table-header'
  | 'keep-image-with-caption'
  | 'keep-equation-with-explanation'
  | 'rebalance-columns'
  | 'prefer-stable-breaks';
```

示例映射：

- 标题孤立 -> `keep-heading-with-next`
- 页尾留白过大 -> `move-next-content-back`
- 段落短尾 -> `move-current-tail-forward`
- 表格整块跳页 -> `split-table-by-row`
- 表格跨页难读 -> `repeat-table-header`
- 多栏不均衡 -> `rebalance-columns`
- 页码跳动过大 -> `prefer-stable-breaks`

## 8. 训练样本

用户反馈要转换成训练样本，而不是直接写死断点。

建议结构：

```ts
interface PaginationTrainingSample {
  sampleId: string;
  sourceBreakId: string;
  documentFingerprint: string;
  features: PaginationBreakFeatures;
  label: PaginationFeedbackLabel;
  weight: number;
  scope: 'current-document' | 'global-preference';
  createdAt: string;
}

interface PaginationBreakFeatures {
  pageFillRatio: number;
  remainingHeightRatio: number;
  beforeBlockType: string;
  afterBlockType: string;
  beforeLineCount?: number;
  afterLineCount?: number;
  isHeadingAtPageEnd: boolean;
  isTableSplit: boolean;
  isImageNearBreak: boolean;
  isEquationNearBreak: boolean;
  columnCount: number;
  columnFillDelta?: number;
}

interface PaginationFeedbackLabel {
  verdict: PaginationBreakVerdict;
  problemTypes: PaginationProblemType[];
  preferredAction?: PaginationPreferredAction;
}
```

样本权重建议：

- `正确`：低到中等权重，用于保护系统已经做对的偏好。
- `不正确 + 高严重度`：高权重，用于强烈调整分页评分。
- `不确定`：低权重，只作为观察数据。
- `current-document`：只影响当前文档或当前项目。
- `global-preference`：进入长期偏好库，但需要多次出现后才提升权重。

## 9. 评分权重

训练结果不应该是黑盒模型，第一版建议输出可解释权重。

```ts
interface PaginationScoringWeights {
  blankSpacePenalty: number;
  headingOrphanPenalty: number;
  paragraphShortTailPenalty: number;
  nextPageShortHeadPenalty: number;
  tableJumpPenalty: number;
  tableReadabilityPenalty: number;
  imagePositionPenalty: number;
  equationPositionPenalty: number;
  columnBalancePenalty: number;
  pageStabilityPenalty: number;
  overflowPenalty: number;
}
```

调整规则示例：

- 多次标记“页尾留白过大” -> 降低过早换页倾向，提高向前填充权重。
- 多次标记“标题孤立” -> 提高标题挂靠惩罚。
- 多次标记“段落短尾” -> 提高短尾惩罚，优先回退自然断点。
- 多次标记“表格整块跳页” -> 提高表格按行拆分收益。
- 多次标记“表格跨页难读” -> 提高表头重复和整行保护权重。
- 多次标记“页码跳动过大” -> 提高稳定性惩罚，优先保持已有断点。

## 10. 自动优化算法

第一版不建议使用神经网络。推荐使用可解释的参数搜索：

1. 从当前权重开始。
2. 根据用户反馈生成候选权重。
3. 每组候选权重重新分页。
4. 对新分页结果自动评分。
5. 选择综合损失最低的一组。
6. 生成优化前后对比。
7. 用户确认后应用。

综合损失可以这样计算：

```ts
loss =
  incorrectBreakPenalty
  + unresolvedProblemPenalty
  + overflowPenalty
  + excessiveBlankSpacePenalty
  + pageCountChangePenalty
  + pageJumpPenalty
  + performancePenalty;
```

候选权重生成策略：

- 对用户明确标记的问题，提高对应惩罚权重。
- 对用户标记为正确的分页点，保护产生该分页点的评分倾向。
- 每次只小幅调整权重，避免分页结果大范围跳动。
- 如果优化后新问题更多，自动回退到上一版权重。

## 11. 审核 UI 设想

后续 UI 可以做成“分页点审核列表”：

- 左侧显示页码和分页点编号。
- 中间显示分页点前后内容摘要。
- 右侧显示系统预判问题和用户反馈控件。

最小控件：

- `正确`
- `不正确`
- `不确定`
- 问题类型多选
- 严重度：`轻微`、`中等`、`严重`
- 影响范围：`仅当前文档`、`作为长期偏好`
- `应用自动优化`
- `查看优化前后`
- `回退本次优化`

注意：

审核列表可以支持“每个分页点都显示”，满足精细审核；也可以默认只突出高风险分页点，减少用户负担。

## 12. 存储分层

建议后续分三层存储：

1. 运行时审核对象
   当前分页后临时生成，不写入 `.layout`。

2. 当前文档反馈
   只影响当前 `.layout` 或当前项目，适合处理文档特例。

3. 用户长期偏好
   存在本机用户配置中，用于后续文档默认优化。

本设计小步不新增实际存储字段。后续实现前需要再次确认：

- 当前文档反馈是否写入 `.layout`。
- 长期偏好是否写入工作区配置。
- 是否需要导入导出偏好配置。

## 13. 与现有系统关系

### 13.1 与 `max-fill-v1`

当前唯一内置分页算法仍是 `max-fill-v1`。

反馈训练第一阶段只影响“断点评分参数”和“问题诊断”，不重写整个分页引擎。

### 13.2 与隐藏测量层

隐藏测量层提供真实块高、真实换行索引和后续可能的真实分页参考。

反馈训练应该优先消费这些测量结果，避免用户反馈被纯估算误差污染。

### 13.3 与多栏分页

多栏分页需要额外记录栏位信息：

- 当前分页点位于第几栏。
- 当前页各栏填充差异。
- 文本是否从当前栏续到下一栏或下一页。

多栏问题应独立标记为 `column-unbalanced`，避免和普通页尾留白混淆。

### 13.4 与导出一致性

如果预览分页和导出结果不一致，用户反馈可能会训练到错误目标。

因此后续训练实现前，应优先保证：

- 画布预览与 HTML/PDF 导出使用同一分页结果。
- 导出前高保真对照只作为校验或补充样本，不直接替代用户反馈。

## 14. 安全边界

必须避免以下错误方向：

- 不让用户手工拖动或钉死每个分页点作为主要训练方式。
- 不把一次文档特例直接升级为全局偏好。
- 不让黑盒模型在实时编辑中直接决定分页。
- 不在用户未确认时自动覆盖已接受的分页偏好。
- 不把训练样本写回正文块，避免污染 `LayoutDocument`。

## 15. 后续实施建议

建议后续拆成 5 个小步：

1. `PH2-09A-feedback-break-model-v1`
   实现分页点审核对象生成，不做 UI。

2. `PH2-09B-feedback-diagnosis-v1`

3. `PH2-09C-feedback-review-ui-v1`
   实现分页点审核列表和用户反馈采集。

5. `PH2-09E-feedback-preference-storage-v1`
   实现当前文档偏好与长期偏好的分层存储。

每一步都需要先确认边界，再写入结构化进度源。

## 16. 当前结论

分页训练算法的第一版应采用“用户反馈驱动的可解释权重优化”，而不是让用户手工排版，也不是直接引入黑盒模型。

用户负责判断：

- 这个分页点是否正确。
- 如果不正确，是什么问题。

系统负责：

- 把问题转成训练样本。
- 自动调整分页评分参数。
- 重新分页。
- 展示优化前后差异。
- 允许回退。

这条路线更符合当前项目的结构化文档模型、分页测试算法1、隐藏测量层和进度治理方式，也更适合后续逐步落地。
