# AGENT 协作约定

## 1. 目标

本项目的开发推进必须同时满足两件事：

1. 按 `LAYOUT3.0产品开发文档.md`、`功能设计及需求.md`、`结构化文档模型V1方案.md` 推进真实开发工作。
2. 按结构化进度源同步维护开发进度，确保每一步推进都有记录、可回溯、可校验。

## 2. 进度文件分工

### 2.1 唯一真相源

以下文件是进度跟踪的唯一真相源，必须手工维护：

- `progress/config.json`
- `progress/tasks/*.json`
- `progress/risks.json`
- `progress/tech-debts.json`
- `progress/logs.json`

### 2.2 自动生成产物

以下文件是展示层产物，禁止手工直接修改：

- `项目开发进度.md`
- `当前执行看板.md`

它们必须通过以下命令生成：

- `npm run progress:build`
- `npm run progress:check`

## 3. 核心执行规则

1. 每次完成一个可独立验收的最小推进步骤，必须立即更新结构化进度源。
2. 未更新结构化进度源并重新生成展示文档的工作，不视为该步骤已完成。
3. 每个开发步骤都必须绑定明确的任务 ID；任务 ID 以 `PHx-xx` 为主，项目治理类记录可使用 `GOV-xxx`。
4. 进度更新至少包括：对应任务状态、必要时的阶段/里程碑/模块状态、以及一条新的进度日志。
5. 如果工作处于进行中但尚未达到完成标准，只能标记为 `进行中`，不能提前写成 `已完成`。
6. 如果出现阻塞、返工、范围变化，必须同步更新 `progress/risks.json`，说明原因、影响范围和下一步处理方式。
7. 前端开发必须使用中文，特别是按钮、功能、描述等，除非是特定专有名词。
8. 代码必须写中文注释，我是代码小白，需要靠注释理解代码。
9. 每确定下一步开发计划后，必须先让我确认方案和细节，然后才能开始改代码。
10. 每次确认下一步开发方案后，开始改代码前，必须先把该步边界写回结构化进度源，至少明确三件事：`本步纳入`、`本步明确不纳入`、`后续归属任务`，避免后续范围漂移或遗忘“这一步没做什么”。
11. 每次完成结构化进度源更新后，必须执行 `npm run progress:build`；提交前建议再执行 `npm run progress:check`。
12.每个任务要充分考虑到后续分页算法等其他关联功能的影响，并提出方案

## 4. “推进一步”判定标准

只有满足以下条件，才算推进了一步并需要登记：

- 有明确对象：功能、模块、文档、测试、配置、接口或流程。
- 有明确产出：代码、文档、测试、配置或可验证结果。
- 有明确验证：能通过运行、测试、检查或审阅确认。

不满足验收条件的中间过程，不得记为完成，只能更新为 `进行中`。

## 5. 标准执行动作

每次开展开发任务时，按以下顺序执行：

1. 先在 `progress/tasks/*.json` 中定位对应阶段、任务、模块和里程碑；如果没有任务卡，先补任务卡。
2. 明确下一步开发计划后，先让我确认方案和细节。
3. 用户确认方案后，先回写该步边界到结构化进度源，至少写清 `本步纳入`、`本步明确不纳入`、`后续归属任务`。
4. 完成当前最小步骤的开发、修复、验证或文档更新。
5. 回写结构化进度源：
   - 更新对应任务卡状态；
   - 必要时更新 `progress/config.json` 中的阶段、里程碑、模块状态；
   - 若有风险或范围变化，更新 `progress/risks.json`；
   - 在 `progress/logs.json` 追加一条新日志。
6. 执行 `npm run progress:build` 更新 `项目开发进度.md` 与 `当前执行看板.md`。
7. 必要时执行 `npm run progress:check` 确认生成产物与真相源一致。
8. 再进入下一个最小步骤。

## 6. 记录要求

每条新增进度日志必须包含以下信息：

- 日期
- 任务 ID
- 所属阶段或里程碑
- 完成的具体步骤
- 动作类型
- 实际结果
- 验证方式
- 影响文件
- 下一步

## 7. 优先级约束

默认按以下顺序推进：

1. 优先完成 `P0 / 第一阶段 / M1` 主链路。
2. 再推进 `P1 / 第二阶段 / M2`。
3. 之后再进入 `P2 / 第三、四阶段 / M3、M4、GA`。

若临时偏离该顺序，必须在日志或风险区写明原因。

## 8. 文档维护约束

1. 如果需求文档、开发文档或阶段基线发生变化，需要同步检查 `progress/config.json`、任务卡、风险和日志是否需要调整。
2. `项目开发进度.md` 与 `当前执行看板.md` 只允许通过脚本生成，不允许直接手改。
3. 如果 `progress:check` 失败，说明结构化真相源与展示层产物不一致，必须先修复再继续。

从本文件生效起，后续每完成一步开发推进，都必须先更新结构化进度源，再生成进度文档。
