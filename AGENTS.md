# AGENT 协作约定

## 1. 目标

本项目的开发推进必须同时满足两件事：

1. 按 `LAYOUT3.0产品开发文档.md` 与 `功能设计及需求.md` 推进真实开发工作。
2. 按 `项目开发进度.md` 同步维护开发进度，确保每一步推进都有记录。

## 2. 核心执行规则

1. 每次完成一个可独立验收的最小推进步骤，必须立即更新 `项目开发进度.md`。
2. 未更新 `项目开发进度.md` 的工作，不视为该步骤已完成。
3. 进度更新至少包括：对应步骤状态、相关模块状态、必要时的里程碑状态、以及一条新的进度日志。
4. 如果工作处于进行中但尚未达到完成标准，只能标记为 `进行中`，不能提前写成 `已完成`。
5. 如果出现阻塞、返工、范围变化，必须同步写入 `项目开发进度.md` 的风险与阻塞区，说明原因和下一步处理方式。
6. 前端的开发必须用中文，特别是按钮、功能、描述等，除非是特顶名词
7. 代码必须写中文注释（我是一个代码小白，你不写注释我看不懂）
8. 每确定下一步开发计划后，必须先让我确定方案和细节后才开始改代码

## 3. “推进一步”判定标准

只有满足以下条件，才算推进了一步并需要登记：

- 有明确对象：功能、模块、文档、测试、配置、接口或流程。
- 有明确产出：代码、文档、测试、配置或可验证结果。
- 有明确验证：能通过运行、测试、检查或审阅确认。

不满足验收条件的中间过程，不得记为完成，只能更新为 `进行中`。

## 4. 标准执行动作

每次开展开发任务时，按以下顺序执行：

1. 先在 `项目开发进度.md` 中定位对应阶段、步骤、模块和里程碑。
2. 完成当前最小步骤的开发、修复、验证或文档更新。
3. 回写 `项目开发进度.md`：更新状态表，并在“进度日志”追加一条记录。
4. 再进入下一个最小步骤。

## 5. 记录要求

每条新增进度日志必须包含以下信息：

- 日期
- 所属阶段或里程碑
- 完成的具体步骤
- 动作类型
- 实际结果
- 验证方式
- 影响文件
- 下一步

## 6. 优先级约束

默认按以下顺序推进：

1. 优先完成 `P0 / 第一阶段 / M1` 主链路。
2. 再推进 `P1 / 第二阶段 / M2`。
3. 之后再进入 `P2 / 第三、四阶段 / M3、M4、GA`。

若临时偏离该顺序，必须在 `项目开发进度.md` 的日志或风险区写明原因。

## 7. 文档维护约束

如果需求文档或开发文档发生变化，需要同步检查 `项目开发进度.md` 是否需要调整基线、阶段拆解、里程碑或完成标准。

从本文件生效起，后续每完成一步开发推进，都必须写入 `项目开发进度.md`。
